RELEASE PLEASE SETUP REFERENCE
==============================

Release Please automates version bumping and changelog generation
based on Conventional Commits.

CONFIG FILE (.github/release-please-config.json)
-------------------------------------------------

{
  "release-type": "node",          // "node", "python", "go", "simple", etc.
  "packages": {
    ".": {
      "release-type": "node",
      "package-name": "my-project",
      "version-file": "package.json",
      "changelog-path": "CHANGELOG.md",
      "extra-files": []            // Additional files with version strings
    }
  },
  "plugins": []                     // None by default
}

MANIFEST FILE (.release-please-manifest.json)
----------------------------------------------

{
  ".": "1.2.3"
}

The manifest tracks the current release version for each package.
Release Please updates this file after each release.

VERSION FILE UPDATES
--------------------

Release Please automatically updates these files depending on release-type:

release-type: "node"
  - package.json (version field)
  - npm-shrinkwrap.json / package-lock.json

release-type: "python"
  - setup.py, setup.cfg, or pyproject.toml (version field)

release-type: "go"
  - version.go (version variable)

release-type: "simple"
  - version.txt

Extra custom files:
  {
    "extra-files": ["src/version.ts", "gradle.properties"]
  }

For files containing version strings not in standard locations, use
extra-files and ensure the version is in a recognizable format.

CHANGELOG GENERATION
--------------------

Release Please generates CHANGELOG.md automatically:

  ## [1.1.0] - 2024-11-15

  ### Features
  * Add user export endpoint (#142) ([abc1234])
  * Add email notification service (#145) ([def5678])

  ### Bug Fixes
  * Fix memory leak in WebSocket handler (#141) ([ghi9012])

  ### ⚠ BREAKING CHANGES
  * Rate limit response format changed - return Retry-After header
    instead of custom X-RateLimit-Reset header. ([jkl3456])

Grouped by Conventional Commit type:
  - feat       -> Features
  - fix        -> Bug Fixes
  - perf       -> Performance Improvements
  - deps       -> Dependencies
  - BREAKING   -> ⚠ BREAKING CHANGES (any type with BREAKING CHANGE footer)

GITHUB ACTION WORKFLOW (.github/workflows/release-please.yml)
--------------------------------------------------------------

name: Release Please

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

jobs:
  release-please:
    runs-on: ubuntu-latest
    steps:
      - uses: googleapis/release-please-action@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          release-type: node
          config-file: .github/release-please-config.json
          manifest-file: .release-please-manifest.json

Behavior:
  1. On push to main, scans commits since last release.
  2. Creates/updates a release PR with version bump + changelog.
  3. When release PR merges, tags the commit and creates a GitHub Release.
  4. Updates the manifest file.

MONOREPO HANDLING
-----------------

For monorepos with multiple packages:

{
  "release-type": "node",
  "packages": {
    "packages/backend": {
      "package-name": "backend"
    },
    "packages/frontend": {
      "package-name": "frontend"
    },
    "packages/shared": {
      "package-name": "shared"
    }
  },
  "plugins": [
    "node-workspace",     // Use npm/yarn workspaces to resolve inter-deps
    "linked-versions"     // Keep linked packages at same version
  ]
}

Each package gets its own release PR and manifest entry.
Use "node-workspace" plugin for npm/yarn workspaces.
Use "linked-versions" to keep certain packages in lockstep.

PRERELEASE CHANNELS
-------------------

Release Please supports prerelease channels via release branches:

on:
  push:
    branches: [main, beta, alpha]

For branch "beta", releases are tagged with -beta.X suffix.
For branch "alpha", releases are tagged with -alpha.X suffix.

Configuration per branch:
  // .github/release-please-config.json
  {
    "channels": {
      "main": { "prerelease": false },
      "beta": { "prerelease": true, "prerelease-type": "beta" },
      "alpha": { "prerelease": true, "prerelease-type": "alpha" }
    }
  }

Each channel maintains its own manifest file:
  .release-please-manifest.json    (main)
  .release-please-manifest-beta.json  (beta)
  .release-please-manifest-alpha.json (alpha)

RELEASE TYPES AVAILABLE
-----------------------

  - "node"     (npm/node - package.json)
  - "python"   (PyPI - setup.py, setup.cfg, pyproject.toml)
  - "go"       (Go modules - version.go)
  - "java"     (Maven - pom.xml)
  - "rust"     (Cargo/Cargo.toml)
  - "simple"   (generic - version.txt)
  - "helm"     (Helm chart - Chart.yaml)
  - "terraform-module" (Terraform module - terraform/)
  - "terraform-provider" (Terraform provider - terraform/)
  - "ocaml"    (opam/dune-project)
  - "elixir"   (mix.exs)
  - "dart"     (pubspec.yaml)
  - "ruby"     (gem)

TROUBLESHOOTING
---------------

- "No commits found since last release": Check that tags are pushed to remote.
- "version_file not found": Verify the release-type matches your project.
- CHANGELOG not generated: Ensure conventional commit format is followed strictly.
  "feat: description" works. "feature: description" does NOT.
- Monorepo release PR not opening: Check that package-specific paths are correct.
- Wrong version bump: Look for "BREAKING CHANGE:" footer in commits.
